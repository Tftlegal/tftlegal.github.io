⚠️ *Внимание: Основной LLM-провайдер недоступен. Использован резервный маршрут!*

🧠 Мульти-Агентный ответ:

Проектирование отказоустойчивого (High Availability, HA) кластера Langfuse — это комплексная задача, так как платформа состоит из нескольких взаимосвязанных сервисов. Основной подход — это **развертывание в Kubernetes с использованием официального Helm-чарта** и замена всех внутренних компонентов (баз данных, кэша) на управляемые отказоустойчивые сервисы вашего облачного провайдера.

Ниже приведена пошаговая архитектурная схема для создания такого кластера.

### 🏗️ Шаг 1: Понимание архитектуры Langfuse

Кластер Langfuse v3 состоит из следующих ключевых компонентов:

*   **Прикладные контейнеры**:
    *   **Langfuse Web**: Основное веб-приложение, UI и API.
    *   **Langfuse Worker**: Фоновый воркер для асинхронной обработки событий, очередей и выполнения задач.
*   **Хранилища данных**:
    *   **PostgreSQL**: Основная транзакционная БД (пользователи, проекты, API-ключи, промпты).
    *   **ClickHouse**: Высокопроизводительная OLAP-БД для хранения всех данных трассировки (трейсы, наблюдения, оценки).
    *   **Redis/Valkey**: Используется как брокер очередей (BullMQ) и для кэширования.
    *   **S3/Blob Storage**: Объектное хранилище для сырых событий и больших вложений (изображения, аудио).

### 🔧 Шаг 2: Обеспечение отказоустойчивости для каждого компонента

Ключевой принцип — каждый компонент должен быть избыточным.

*   **Прикладной слой (Web & Worker)**: Запускайте минимум **2 реплики** каждого контейнера для обеспечения отказоустойчивости. Распределите их по разным зонам доступности (AZ) в вашем кластере Kubernetes.
    *   **Worker-воркеры**: Масштабируйте их горизонтально в зависимости от нагрузки. Основной метрикой для автоскейлинга является загрузка CPU (порог > 50% для контейнера с 2 vCPU) и длина очереди (`langfuse.queue.ingestion.length`).

*   **Слой баз данных (Критически важно!)**:
    *   **PostgreSQL**: **Не используйте** встроенный в чарт PostgreSQL для продакшена. Используйте **управляемый сервис** (например, AWS RDS, GCP Cloud SQL, Azure Database for PostgreSQL) с включенным **Multi-AZ** режимом для автоматического переключения при сбоях.
    *   **ClickHouse**: Аналогично, используйте **управляемый ClickHouse Cloud** или **ClickHouse с репликацией** на уровне Kubernetes (например, используя оператор). Важное ограничение: Langfuse **не поддерживает многошардовый (multi-shard) кластер**, значение `CLICKHOUSE_CLUSTER_NAME` должно быть `1`. Для отказоустойчивости используйте реплики, а не шарды.
    *   **Redis/Valkey**: Используйте **управляемый сервис** (например, AWS ElastiCache, GCP Memorystore) с включенной **репликацией (Replication Group)** и автоматическим переключением при сбоях.

*   **Объектное хранилище (S3)**:
    *   Используйте **управляемое объектное хранилище** вашего облачного провайдера (AWS S3, GCS, Azure Blob Storage), которое по умолчанию является отказоустойчивым и реплицируется внутри региона.

### 🚀 Шаг 3: Развертывание с помощью Terraform и Helm

Самый надежный способ развернуть отказоустойчивый кластер — это комбинация **Terraform** для создания инфраструктуры и **Helm** для развертывания самого приложения.

1.  **Подготовка инфраструктуры с Terraform**:
    *   Используйте официальные [Terraform-модули от Langfuse](https://github.com/langfuse) для AWS, GCP или Azure. Эти модули развертывают **высокодоступные кластеры по умолчанию** и используют нативные управляемые сервисы (RDS, ElastiCache, S3).

2.  **Развертывание приложения с Helm**:
    *   Используйте официальный [Helm-чарт langfuse-k8s](https://github.com/langfuse/langfuse-k8s).
    *   В файле `values.yaml` укажите **внешние эндпоинты** для созданных вами управляемых баз данных (PostgreSQL, ClickHouse, Redis) и S3-хранилища, отключив тем самым установку внутренних неотказоустойчивых версий этих сервисов.

### ⚠️ Шаг 4: Критические предостережения при настройке

*   **Миграции БД (Критично!)**: При запуске или обновлении кластера **убедитесь, что запущена только одна реплика `langfuse-web`**. Если во время миграций схемы данных будет запущено несколько реплик, это может привести к **взаимоблокировкам (deadlocks) и сбоям миграций** из-за гонок Prisma. После завершения миграций можно снова увеличить число реплик.
*   **Единый часовой пояс**: **Все** компоненты инфраструктуры (ClickHouse, PostgreSQL) **обязаны** работать с часовым поясом **UTC**. Использование другого пояса приведет к некорректной работе запросов и пустым результатам.
*   **Мульти-регион**: Запуск кластера в **мульти-региональном** режиме (например, AWS us-east-1 и eu-west-1) является **сложной задачей** и официально не рекомендуется из-за задержек и сложности синхронизации данных. Вместо этого используйте **мульти-зону доступности (Multi-AZ)** в рамках одного региона.

### 🛡️ Шаг 5: Резервное копирование и мониторинг

*   **Резервное копирование**: Внедрите стратегию резервного копирования для каждого компонента.
    *   **PostgreSQL и ClickHouse**: Используйте встроенные механизмы бэкапов управляемых сервисов (PITR).
    *   **Для самовыраженного ClickHouse**: Настройте автоматические снапшоты томов (Volume Snapshots) в Kubernetes или используйте Velero для бэкапа всего кластера.
*   **Мониторинг**: Настройте мониторинг для всех компонентов. Ключевые метрики:
    *   Длина очереди (`langfuse.queue.ingestion.length`) для масштабирования Worker.
    *   Использование CPU/Памяти для Web и Worker подов.
    *   Состояние реплик баз данных и задержки репликации.

### 💎 Итоговая архитектурная схема

В конечном итоге, ваш отказоустойчивый кластер Langfuse должен выглядеть так:

1.  **Kubernetes кластер** (EKS, GKE, AKS), распределенный по **нескольким зонам доступности (AZ)**.
2.  **Приложение Langfuse** (Web и Worker) запущено в виде **минимум 2 реплик** каждая.
3.  **Все базы данных** (PostgreSQL, ClickHouse, Redis) являются **внешними управляемыми сервисами** с включенной репликацией и автоматическим переключением (Multi-AZ).
4.  **Объектное хранилище** (S3) используется как надежное, реплицируемое хранилище для событий и файлов.
5.  Настроены **автоматические резервные копии** и **мониторинг** всех компонентов.
