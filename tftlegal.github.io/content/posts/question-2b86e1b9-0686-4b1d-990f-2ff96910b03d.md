# Полное руководство по настройке PostgreSQL: от базовых параметров до продвинутых практик

PostgreSQL «из коробки» поставляется с крайне консервативными настройками, рассчитанными на то, чтобы база данных запускалась даже на самом слабом железе. Эти параметры совершенно не подходят для production-серверов с большими объёмами памяти и мощными процессорами. В этой статье мы разберём все ключевые аспекты настройки PostgreSQL — от выбора стратегии до конкретных цифр и лучших практик.

---

## 1. OLTP или OLAP? Определяем тип нагрузки

Перед настройкой любого параметра необходимо чётко понять, с каким типом нагрузки вы работаете. От этого зависят все дальнейшие решения.

**OLTP (Online Transaction Processing)** — это операционная, повседневная работа. Тысячи мелких операций INSERT и UPDATE, каждая из которых должна выполняться за миллисекунды. Данные изменяются по одной строке за раз. Основная цель — быстрая фиксация результата каждой транзакции.

**OLAP (Online Analytical Processing)** — это аналитика и отчёты. Сложные запросы, обрабатывающие миллионы и миллиарды строк. Операций мало, но каждая из них может выполняться минуты или даже часы. Данные только читаются (SELECT), агрегируются, суммируются. Основная цель — поиск закономерностей и трендов.

Реляционные СУБД, включая PostgreSQL, исторически больше рассчитаны на OLTP-нагрузку, но при правильной настройке способны эффективно работать и с аналитикой.

## 2. Инструменты для расчёта параметров

Вместо того чтобы подбирать значения вручную, удобно использовать специализированные конфигураторы:

- **pgtune.leopard** — https://pgtune.leopard.in.ua/ (работает только через VPN)
- **pgconfigurator.cybertec** — https://pgconfigurator.cybertec.at/
- **pgconfig** — https://www.pgconfig.org/
- **pgtune.sainth** — https://pgtune.sainth.de/

Эти инструменты учитывают объём RAM, количество ядер CPU, тип дисков и предполагаемую нагрузку, выдавая готовый набор параметров для `postgresql.conf`.

## 3. Настройка параметров CPU и параллелизма

PostgreSQL умеет распараллеливать выполнение запросов и служебных операций. Важно правильно настроить иерархию параметров, отвечающих за параллелизм.

### Иерархия параметров параллелизма

```
max_worker_processes (общий пул всех фоновых процессов)
  ↳ max_parallel_workers (сколько из общего пула можно взять для параллельных операций)
      ↳ max_parallel_workers_per_gather (сколько на один запрос)
      ↳ max_parallel_maintenance_workers (сколько на одну служебную команду)
```

### Ключевые параметры

| Параметр | Рекомендация | Пояснение |
|----------|--------------|-----------|
| **max_worker_processes** | Число ядер CPU или чуть выше (+20%) | Общий лимит всех фоновых процессов в кластере. Устанавливать выше числа ядер обычно бессмысленно. |
| **max_parallel_workers** | Равен max_worker_processes | Сколько процессов из общего пула можно использовать для параллельных операций. Не может быть больше max_worker_processes. |
| **max_parallel_workers_per_gather** | Ядра / 2 (но не выше 4-6 для OLTP) | Максимум рабочих процессов на один запрос. Для OLAP можно поднимать выше. |
| **max_parallel_maintenance_workers** | 4-6 | Параллелизм для CREATE INDEX, VACUUM FULL, ALTER TABLE. Выше 6 редко даёт прирост из-за конкуренции за дисковую подсистему. |

## 4. Настройка памяти: Shared Memory

PostgreSQL использует разделяемую память (Shared Memory) для кеширования данных. Это один из самых важных блоков настроек.

### shared_buffers

**Назначение**: объём памяти для кеширования таблиц и индексов в разделяемой памяти.

**Рекомендации**:
- **25–40% от общего объёма RAM**
- Для серверов с 128+ GB RAM долю можно снизить до 15–20%, так как остальная память будет использоваться ОС для кеширования файлов
- Для систем с памятью выше 32 GB часто рекомендуют фиксированные 8 GB

**Пример**: для сервера с 32 GB RAM → `shared_buffers = 8–10 GB`

### effective_cache_size

**Назначение**: ориентир для планировщика запросов. Это **не выделяемая память**, а подсказка PostgreSQL о том, сколько всего кеша (shared_buffers + кеш файловой системы ОС) примерно доступно.

**Рекомендации**:
- **50–75% от общего объёма RAM**
- **Важно**: `shared_buffers + effective_cache_size` не должны превышать объём RAM. Нужно оставлять запас для ОС.

**Пример**: для сервера с 32 GB RAM → `effective_cache_size = 20–22 GB`

## 5. Настройка памяти: Backend Memory (на подключение)

Каждое подключение к PostgreSQL потребляет собственную память. Именно поэтому PostgreSQL не любит большого количества подключений — каждый новый бэкенд-процесс забирает ресурсы.

### work_mem

**Назначение**: память для сортировок, хеш-соединений и других внутренних операций в рамках одного запроса.

**Критическое понимание**: `work_mem` выделяется **на одну операцию** (сортировку, хеш-соединение), а не на одно подключение. В одном запросе может быть несколько таких операций.

**Рекомендации**:

| Сценарий | Рекомендуемое значение |
|----------|----------------------|
| OLTP (много подключений) | 4–16 MB |
| OLAP (мало подключений, сложные запросы) | 64–256 MB и выше |

**Формула для OLTP**: `work_mem ≈ (RAM - shared_buffers - запас) / (max_connections * среднее_число_операций_в_запросе)`

**Опасность**: если выставить `work_mem = 256 MB` при 100 подключениях, и каждое выполнит запрос с двумя сортировками, потенциальное потребление составит `100 × 2 × 256 MB = 51.2 GB` — это гарантированно вызовет OOM Killer на сервере с 32 GB RAM.

### temp_buffers

**Назначение**: буферный кеш для временных таблиц в рамках одного подключения.

**Рекомендация**: обычно достаточно значения по умолчанию (8 MB). Повышать до 64–128 MB стоит только при активной работе с временными таблицами, подтверждённой мониторингом.

### maintenance_work_mem

**Назначение**: память для служебных операций — VACUUM, CREATE INDEX, REINDEX, ALTER TABLE ADD FOREIGN KEY, CLUSTER.

**Ключевое отличие от work_mem**: эта память выделяется не на каждое подключение, а на конкретный процесс обслуживания. Таких процессов одновременно может быть несколько.

**Рекомендации**:
- **256 MB – 1 GB** для большинства систем
- **5–10% от общего объёма RAM** для больших серверов
- Для серверов с 64–128 GB RAM и большими таблицами — **4–8 GB**

**Зачем увеличивать**:
- VACUUM работает быстрее и эффективнее, меньше вытесняет данные на диск
- CREATE INDEX для больших таблиц ускоряется в разы

## 6. Настройка дискового ввода-вывода

### random_page_cost

**Назначение**: стоимость случайного чтения с диска относительно последовательного. Влияет на выбор планировщика между индексным сканированием и полным сканированием таблицы.

**Исторический контекст**: значение по умолчанию 4.0 было выбрано во времена механических HDD, когда случайное чтение было значительно дороже последовательного.

**Рекомендации**:

| Тип диска | Рекомендуемое значение |
|-----------|----------------------|
| HDD | 4.0 (по умолчанию) |
| SAS-диски | 1.25 – 2.0 |
| SSD | 1.1 – 1.5 |
| Данные полностью в кеше | 1.0 |

**Важно**: не стоит устанавливать это значение слишком высоким (например, 30) на основе бенчмарков — это может навредить глобальной производительности системы, так как модель затрат PostgreSQL не учитывает кеширование и конкурентность.

### effective_io_concurrency

**Назначение**: количество параллельных операций ввода-вывода, которое может выполнять одна сессия.

**Рекомендации**:
- HDD: 1–2
- SSD: 100–200

### maintenance_io_concurrency

Аналог `effective_io_concurrency`, но для служебных операций (VACUUM, CREATE INDEX). Для SSD выставляется в 100–200.

### fsync

**Назначение**: принудительный сброс данных журнала на диск.

**Рекомендация**: **всегда ON** в production. Это главная защита от потери данных при сбое питания. Отключение допустимо только временно при загрузке огромных объёмов данных (например, pg_restore).

### synchronous_commit

**Назначение**: момент подтверждения транзакции для пользователя.

**Рекомендация**: **ON** для надёжности, **OFF** — для производительности с риском потери данных при сбое.

### full_page_writes

**Рекомендация**: **ON** (защита от порчи данных из-за частичной записи страниц).

### checkpoint_completion_target

**Рекомендация**: **0.9** — распределяет нагрузку от checkpoint на 90% интервала, снижая пиковые нагрузки на I/O.

### vm.swappiness (на уровне ОС)

**Рекомендация**: **1 или 0**. Это заставляет ядро Linux использовать swap только в самых критических ситуациях. Полностью отключать swap нельзя — это риск срабатывания OOM Killer.

## 7. Настройки оптимизатора

| Параметр | Рекомендация | Пояснение |
|----------|--------------|-----------|
| **join_collapse_limit** | Не трогать для OLTP | Для OLAP можно увеличить, но планировщик будет работать дольше |
| **default_statistics_target** | Не трогать для OLTP | Для OLAP можно увеличить для более точной статистики |
| **track_activity_query_size** | 1 MB | Не влияет на производительность, но полезен для мониторинга длинных запросов |
| **geqo** | OFF (для OLTP) | Генетическая оптимизация полезна только для сложных OLAP-запросов |

Все `enable_*` параметры (bitmapscan, hashjoin, indexscan, indexonlyscan, mergejoin, nestloop, seqscan, sort) рекомендуется оставлять в значении **ON**.

## 8. Бенчмаркинг с pgbench

`pgbench` — встроенная утилита для тестирования производительности PostgreSQL.

### Инициализация

```bash
# Создать базу данных
CREATE DATABASE test;

# Инициализировать тестовые таблицы
pgbench -i test
```

Будут созданы 4 таблицы: `pgbench_branches`, `pgbench_tellers`, `pgbench_accounts`, `pgbench_history`.

### Запуск теста

```bash
pgbench -c 50 -j 2 -P 10 -T 60 test
```

| Параметр | Значение | Пояснение |
|----------|----------|-----------|
| `-c` | 50 | Число имитируемых клиентов (одновременных сессий) |
| `-j` | 2 | Число рабочих потоков |
| `-P` | 10 | Отчёт о прогрессе каждые 10 секунд |
| `-T` | 60 | Длительность теста в секундах |

**Самый важный показатель**: **TPS (транзакций в секунду)**. Чем выше — тем лучше.

### Альтернатива: sysbench

Для более реалистичных тестов можно использовать **sysbench-tpcc**:
- https://github.com/Percona-Lab/sysbench-tpcc
- https://github.com/akopytov/sysbench

## 9. Дополнительные Best Practices

### 9.1. Всегда используйте пул соединений

Каждое подключение к PostgreSQL создаёт отдельный бэкенд-процесс. При большом количестве подключений память заканчивается быстрее, чем выполняются запросы.

**Решение**: PgBouncer или аналогичный пулер соединений. Это позволяет держать `max_connections` на разумном уровне (100–200).

### 9.2. Настройка autovacuum

Autovacuum критически важен для поддержания производительности. Он очищает «мёртвые» кортежи, оставшиеся после UPDATE и DELETE.

**Базовые рекомендации**:
- Никогда не отключайте autovacuum
- Для больших таблиц с высокой нагрузкой на запись настраивайте индивидуально:

```sql
ALTER TABLE orders SET (
    autovacuum_vacuum_scale_factor = 0.01,
    autovacuum_analyze_scale_factor = 0.005
);
```

Дефолтные 20% — слишком много для больших таблиц.

### 9.3. Мониторинг — основа настройки

Без мониторинга вы гадаете. С мониторингом — вы принимаете обоснованные решения.

**Что мониторить**:
- **Кеш-хит-ратио**: цель > 95%
- **Временные файлы**: если появляются — значит, `work_mem` слишком мал
- **Активные подключения**: `SELECT count(*) FROM pg_stat_activity WHERE state='active'`
- **Блот (раздувание) таблиц**: отслеживайте через расширение pgstattuple

### 9.4. Индексы: меньше да лучше

- **Неиндексированные таблицы** → полные сканирования → медленные запросы
- **Переиндексированные таблицы** → медленные INSERT/UPDATE, раздувание, медленный VACUUM
- Используйте `pg_stat_all_indexes` для поиска неиспользуемых индексов (`idx_scan = 0`)
- Основной инструмент анализа запросов — `EXPLAIN ANALYZE`

### 9.5. JIT-компиляция

Для сложных аналитических запросов может быть полезна JIT-компиляция (Just-In-Time). Для OLTP-нагрузок JIT обычно не даёт выигрыша и может даже ухудшить производительность из-за накладных расходов на компиляцию.

### 9.6. Используйте huge pages

На системах с большим объёмом RAM использование huge pages может улучшить производительность за счёт снижения накладных расходов на управление страницами памяти.

## 10. Пример конфигурации

Ниже — пример настроек для сервера с **32 GB RAM, 8 ядрами CPU, SSD-дисками** и **OLTP-нагрузкой**:

```ini
# Память
shared_buffers = 8GB
effective_cache_size = 20GB
work_mem = 16MB
maintenance_work_mem = 1GB
temp_buffers = 8MB

# Параллелизм
max_worker_processes = 8
max_parallel_workers = 8
max_parallel_workers_per_gather = 4
max_parallel_maintenance_workers = 4

# Диск
random_page_cost = 1.1
effective_io_concurrency = 200
maintenance_io_concurrency = 200
fsync = on
synchronous_commit = on
full_page_writes = on
checkpoint_completion_target = 0.9

# Соединения
max_connections = 200  # с PgBouncer можно меньше

# Мониторинг
track_activity_query_size = 1MB
shared_preload_libraries = 'pg_stat_statements'
```

## Заключение

Тюнинг PostgreSQL — это не разовое действие, а непрерывный процесс. Начните с базовых настроек (25% RAM на shared_buffers, адекватный work_mem для вашей нагрузки, random_page_cost для SSD), затем:

1. **Внедрите мониторинг** — без данных вы не узнаете, что работает, а что нет
2. **Настройте autovacuum** — профилактика деградации производительности
3. **Используйте пул соединений** — PgBouncer спасёт от исчерпания памяти
4. **Анализируйте медленные запросы** через pg_stat_statements и EXPLAIN
5. **Тестируйте изменения** через pgbench перед применением в production

Помните: оптимальные значения зависят от вашего конкретного оборудования, нагрузки и данных. Используйте рекомендации как отправную точку, но всегда проверяйте на своей системе.
