# PostgreSQL: Внутреннее устройство, конфигурация и управление параметрами
Давайте суссируем всю базовую информацию по настройке постгреса, и научимся использовать системные представления `pg_settings` и `pg_file_settings` для диагностики, а также выясним почему знание иерархии параметров спасает от неожиданных сбоев.

## 1. Postmaster — сердце PostgreSQL

Каждый раз, когда вы запускаете PostgreSQL, первым стартует процесс **postmaster** (в современных версиях исполняемый файл называется просто `postgres`). Это главный управляющий процесс (демон), который становится ядром всей дальнейшей работы. По сути, postmaster — это диспетчер, который:

- **Запускает и останавливает сервер** (команда `pg_ctl start` фактически запускает postmaster).
- **Прослушивает сетевой порт** (по умолчанию 5432) на предмет входящих подключений.
- **Аутентифицирует клиентов**, проверяя `pg_hba.conf` и `pg_ident.conf`.
- **Порождает дочерние процессы** (backend processes) для каждого клиентского подключения — каждый такой процесс обслуживает одну сессию и выполняет все SQL-запросы.
- **Управляет разделяемой памятью** (shared memory), которая используется всеми процессами для буферного кеша и других глобальных структур.
- **Координирует фоновые процессы**: `checkpointer`, `background writer`, `walwriter`, `autovacuum launcher`, `stats collector` и другие.
- **Контролирует восстановление** при некорректной остановке (наличие файлов `standby.signal` или `recovery.signal`).
- **Аварийно останавливает весь кластер**, если критический фоновый процесс завершается с ошибкой — чтобы предотвратить повреждение данных.

Если выполнить на сервере `ps aux | grep postgres`, вы увидите примерно такую картину:

```
postgres  123   /usr/lib/postgresql/16/bin/postgres -D /var/lib/postgresql/17/main/
postgres  456   postgres: checkpointer
postgres  457   postgres: background writer
postgres  458   postgres: walwriter
postgres  459   postgres: autovacuum launcher
postgres  460   postgres: stats collector
postgres  461   postgres: logical replication launcher
postgres  789   postgres: mydb user123 192.168.1.100(54322) idle
postgres  790   postgres: mydb user123 192.168.1.100(54323) SELECT
```

- PID 123 — это главный процесс **postmaster** (хотя в списке он называется `postgres`).
- PID 456–461 — фоновые процессы, порождённые postmaster.
- PID 789, 790 — дочерние backend-процессы, каждый обслуживает отдельное клиентское подключение. В их командной строке указаны база данных, пользователь, хост и текущий статус.

Если на сервере запущено несколько кластеров PostgreSQL (разные версии или разные порты), вы увидите несколько процессов `postgres -D ...` — каждый со своим каталогом данных и своим набором дочерних процессов.

## 2. Конфигурационные файлы: где живут настройки

PostgreSQL читает настройки из нескольких файлов. Главный из них — **postgresql.conf**. Именно он определяет поведение сервера. Расположение этого файла зависит от операционной системы:

- **Ubuntu/Debian**: `/etc/postgresql/<version>/main/postgresql.conf`
- **CentOS/RHEL**: `/var/lib/pgsql-<version>/data/postgresql.conf`
- **macOS**: `/usr/local/var/postgres/postgresql.conf`
- **Windows**: `C:\Program Files\PostgreSQL\<version>\data\postgresql.conf`

Чтобы узнать точный путь к конфигурационному файлу вашего работающего кластера, выполните в psql:

```sql
SHOW config_file;
-- или
SELECT current_setting('config_file');
```

Путь к каталогу данных (data_directory) также можно узнать через:

```sql
SHOW data_directory;
```

### 2.1. postgresql.conf — основной файл

Этот файл содержит сотни параметров, разделённых на логические категории (Connection Settings, Memory, Query Tuning, Replication и т.д.). Синтаксис простой:

```
<имя_параметра> = <значение>
```

- Пробелы и знак «=» необязательны, но рекомендуются для читаемости.
- Комментарии начинаются с `#`.
- Если параметр указан несколько раз, применяется **последнее** значение.
- Единицы измерения: `kB`, `MB`, `GB`, `ms`, `s`, `min` и т.д. (важно не ошибиться).

В конце файла разработчики рекомендуют добавлять свои кастомные настройки в отдельной секции (например, `# CUSTOMIZED OPTIONS`), чтобы упростить их поиск и перенос при обновлениях.

### 2.2. pg_hba.conf — контроль доступа

Файл **pg_hba.conf** (host-based authentication) определяет, кто, откуда и каким способом может подключаться к базам данных, а также — кто может устанавливать соединения для репликации. Правила применяются **сверху вниз**, и используется **первое совпавшее** правило. Поэтому порядок записей критичен.

Пример правил:

```
# Разрешить локальные подключения через Unix-сокет с проверкой по паролю (scram-sha-256)
local   all             all                                     scram-sha-256

# Разрешить подключения с IP 192.168.1.0/24 только пользователю admin
host    all             admin           192.168.1.0/24          scram-sha-256

# Запретить подключения с 10.0.0.5
host    all             all             10.0.0.5/32             reject

# Разрешить репликацию пользователю repl_user с любого адреса
host    replication     repl_user       0.0.0.0/0               scram-sha-256
```

Методы аутентификации:
- `trust` — без пароля (только для доверенных локальных сетей).
- `md5` / `scram-sha-256` — проверка пароля (рекомендуется `scram-sha-256`).
- `peer` — сопоставление с пользователем ОС (только локальные).
- `cert` — проверка SSL-сертификата.
- `reject` — явный запрет.

Важно: псевдоним `all` в столбце DATABASE означает «все базы данных», но **не включает** специальную базу `replication`. Для репликации нужно указывать `replication` явно.

После изменения `pg_hba.conf` **перезагрузка кластера не требуется** — достаточно перечитать конфигурацию:

```sql
SELECT pg_reload_conf();
```

Или через командную строку:

```bash
sudo pg_ctlcluster 17 main reload
# или
sudo systemctl reload postgresql@17-main
```

### 2.3. pg_ident.conf — устаревший механизм

Файл `pg_ident.conf` предназначен для отображения системных имён пользователей ОС на имена пользователей PostgreSQL (механизм Ident Authentication). На практике он используется крайне редко и почти полностью вытеснен методами `peer` и парольной аутентификацией. В современных установках этот файл обычно остаётся нетронутым.

### 2.4. postgresql.auto.conf — автоматический конфиг

Начиная с версии 9.4, PostgreSQL поддерживает файл **postgresql.auto.conf**, который управляется командами `ALTER SYSTEM`. Он находится внутри каталога данных и имеет более высокий приоритет, чем `postgresql.conf`. Это означает, что параметры, записанные в `postgresql.auto.conf`, переопределяют соответствующие настройки из основного файла.

Главные преимущества `postgresql.auto.conf`:
- **Безопасность** — вы не рискуете внести синтаксическую ошибку вручную; команда `ALTER SYSTEM SET` проверяет значение перед записью.
- **Удобство обновлений** — при обновлении версии PostgreSQL системный менеджер пакетов может предложить заменить `postgresql.conf` на новую версию. Ваши кастомные настройки, лежащие в `postgresql.auto.conf`, при этом не затрагиваются.

Изменить параметр через `ALTER SYSTEM`:

```sql
ALTER SYSTEM SET work_mem = '64MB';
ALTER SYSTEM SET max_connections = 300;
```

Сбросить параметр (удалить из `postgresql.auto.conf`):

```sql
ALTER SYSTEM RESET work_mem;
ALTER SYSTEM RESET all;   -- удалить все изменения
```

После выполнения этих команд изменения записываются в файл, но **не применяются автоматически**. Для применения большинства параметров требуется либо перезагрузка (`restart`), либо перечитывание конфигурации (`reload`), в зависимости от контекста (`context`). О том, как определить, что требуется, мы поговорим позже.

## 3. Иерархия применения параметров

Одна из мощных возможностей PostgreSQL — возможность задавать параметры на разных уровнях, причём более высокий уровень переопределяет более низкий. Это позволяет гибко настраивать поведение для конкретных сессий, пользователей или баз данных.

Иерархия (от высшего приоритета к низшему):

1. **Настройки на уровне сессии** (команды `SET`, `SET LOCAL`) — имеют максимальный приоритет и действуют только в рамках текущего подключения или транзакции.
2. **Настройки на уровне функции** (задаются в объявлении функции через `SET`) — действуют на время выполнения функции.
3. **Настройки на уровне базы данных** (`ALTER DATABASE ... SET ...`) — применяются для всех сессий, подключённых к конкретной базе.
4. **Настройки на уровне роли (пользователя)** (`ALTER ROLE ... SET ...`) — применяются для всех сессий данного пользователя (возможно, только в определённой базе).
5. **Настройки на уровне кластера** (из `postgresql.conf`, `postgresql.auto.conf` или командной строки) — базовые значения по умолчанию.

### 3.1. Настройки на уровне сессии

Любой пользователь может изменить параметры, у которых `context = 'user'`, а суперпользователь — параметры с `context = 'superuser'`. Для этого используются команды:

```sql
-- Установить значение до конца сессии
SET work_mem TO '16MB';

-- Установить только для текущей транзакции
BEGIN;
SET LOCAL work_mem = '32MB';
-- ... запросы ...
COMMIT;  -- после COMMIT или ROLLBACK параметр возвращается к прежнему значению
```

Если вы не находитесь внутри транзакции, `SET LOCAL` не сработает (выдаст ошибку). Альтернативный способ — использовать функцию `set_config`:

```sql
SELECT set_config('work_mem', '16MB', false);  -- false = до конца сессии
SELECT set_config('work_mem', '32MB', true);   -- true = только для текущей транзакции (даже вне BEGIN это сработает)
```

Сбросить параметр к значению по умолчанию (на уровне кластера или БД/роли):

```sql
RESET work_mem;
RESET ALL;   -- сбросить все изменённые параметры сессии
```

### 3.2. Настройки на уровне функции

При создании функции можно указать параметры, которые будут действовать только во время её выполнения:

```sql
CREATE OR REPLACE FUNCTION heavy_report()
RETURNS TABLE(...) AS $$
BEGIN
    -- тело функции
END;
$$ LANGUAGE plpgsql
SET work_mem = '256MB'
SET statement_timeout = '5min';
```

Теперь при каждом вызове `heavy_report()` будут временно установлены указанные параметры.

### 3.3. Настройки на уровне базы данных

Изменить параметр для всех сессий, подключающихся к конкретной базе:

```sql
ALTER DATABASE my_analytics SET work_mem = '1GB';
```

Теперь все пользователи, работающие с `my_analytics`, по умолчанию получат `work_mem = 1GB`.

### 3.4. Настройки на уровне роли

Аналогично для конкретного пользователя:

```sql
ALTER ROLE report_user SET statement_timeout = '30min';
```

Можно комбинировать — задать параметр для пользователя только при подключении к определённой БД:

```sql
ALTER ROLE report_user IN DATABASE my_analytics SET work_mem = '2GB';
```

### 3.5. Проверка действующих настроек

Чтобы узнать текущее значение параметра, используйте `SHOW` или `current_setting`:

```sql
SHOW work_mem;
SELECT current_setting('work_mem');
```

Чтобы увидеть, откуда взялось текущее значение, обратитесь к представлению `pg_settings`:

```sql
SELECT name, setting, source, sourcefile, sourceline
FROM pg_settings
WHERE name = 'work_mem';
```

Поле `source` покажет:
- `default` — значение по умолчанию.
- `configuration file` — из `postgresql.conf` или `postgresql.auto.conf`.
- `database` — из `ALTER DATABASE`.
- `user` — из `ALTER ROLE`.
- `session` — из `SET` в текущей сессии.

Важно: `pg_settings` показывает значения **для текущей сессии**. Если вы запросили параметры роли `report_user`, но сами подключены как `postgres`, вы не увидите специфичных для `report_user` значений. Чтобы увидеть настройки, установленные для разных ролей и БД, можно выполнить запрос:

```sql
SELECT
    COALESCE(r.rolname, 'cluster-wide') AS role,
    COALESCE(d.datname, 'cluster-wide') AS database,
    s.setconfig AS settings
FROM pg_db_role_setting s
LEFT JOIN pg_roles r ON r.oid = s.setrole
LEFT JOIN pg_database d ON d.oid = s.setdatabase;
```

Этот запрос покажет все нестандартные настройки, заданные через `ALTER ROLE` и `ALTER DATABASE`.

## 4. Представления pg_settings и pg_file_settings

Для глубокого понимания конфигурации PostgreSQL предоставляет два системных представления. Они дополняют друг друга и помогают ответить на ключевые вопросы: «Какое значение действует сейчас?» и «Что записано в файлах и будет ли это применено?».

### 4.1. pg_settings — активные настройки

`pg_settings` показывает **текущие значения параметров в памяти**, то есть те, которые реально использует работающий сервер. Это представление содержит множество столбцов, но самые важные:

- `name` — имя параметра.
- `setting` — текущее значение.
- `unit` — единица измерения (если есть).
- `context` — уровень, на котором можно изменять параметр:
  - `internal` — нельзя изменить.
  - `postmaster` — требует перезапуска кластера.
  - `sighup` — достаточно перечитать конфиг (`pg_reload_conf()`).
  - `user` / `superuser` — можно менять «на лету» в сессии (SET).
  - `backend` — можно установить только при старте сессии (в строке подключения).
- `source` — источник текущего значения (default, configuration file, session, database, user и т.д.).
- `sourcefile` / `sourceline` — файл и строка, если значение взято из конфига.
- `pending_restart` — `true`, если параметр изменён, но требует перезагрузки для применения.
- `boot_val` — значение по умолчанию, заданное при компиляции.
- `reset_val` — значение, которое будет применено при `RESET` в сессии.

Пример полезного запроса: найти все параметры, которые требуют перезагрузки (`postmaster`):

```sql
SELECT name, setting, short_desc
FROM pg_settings
WHERE context = 'postmaster'
ORDER BY name;
```

Посмотреть, сколько параметров можно изменить без перезагрузки:

```sql
SELECT context, COUNT(*) FROM pg_settings GROUP BY context ORDER BY 2 DESC;
```

### 4.2. pg_file_settings — содержимое конфигурационных файлов

В отличие от `pg_settings`, **pg_file_settings** показывает, что PostgreSQL **прочитал из файлов** на диске. Это не активные значения, а сырые данные из всех конфигурационных файлов (включая `postgresql.conf`, `postgresql.auto.conf`, файлы из `include` и `include_dir`).

Основные столбцы:

- `sourcefile` — путь к файлу.
- `sourceline` — номер строки.
- `name` — имя параметра.
- `setting` — значение, записанное в файле.
- `applied` — будет ли это значение применено при перечитывании конфигурации:
  - `true` — значение корректно и будет применено (или уже применено).
  - `false` — значение по какой-то причине не будет применено (смотреть в поле `error`).
- `error` — текст ошибки, если параметр не был применён.

Это представление незаменимо для диагностики проблем с конфигурацией. Например, вы изменили `shared_buffers` в `postgresql.conf`, выполнили `pg_reload_conf()`, но `SHOW shared_buffers` показывает старое значение. Смотрим:

```sql
SELECT name, setting, applied, error
FROM pg_file_settings
WHERE name = 'shared_buffers';
```

Скорее всего, увидите `applied = false` и `error = 'parameter cannot be changed without restarting the server'`. Это означает, что для применения `shared_buffers` требуется не `reload`, а **restart** кластера.

### 4.3. Сравнение pg_settings и pg_file_settings

| Критерий | pg_settings | pg_file_settings |
|----------|-------------|------------------|
| Что показывает | Текущие активные значения в памяти | Значения, прочитанные из файлов на диске |
| Ключевой вопрос | «Что сейчас работает?» | «Что написано в файлах и применится ли?» |
| Источник | Поле `source` | Поля `sourcefile`, `sourceline` |
| Ошибки | Не показывает | Показывает ошибки в `error` |
| Обновление | Обновляется после `pg_reload_conf()` и `restart` | Обновляется только при чтении файлов (не требует перезагрузки) |
| Применение | Всегда показывает применяемые значения | `applied = false` означает, что значение игнорируется |

### 4.4. Практический кейс: диагностика не применяемого параметра

**Ситуация**: вы установили `max_connections = 300` через `ALTER SYSTEM SET max_connections = 300;`, но `SHOW max_connections` показывает 100.

**Шаг 1**. Проверяем `pg_settings`:

```sql
SELECT name, setting, source, pending_restart
FROM pg_settings
WHERE name = 'max_connections';
```

Если `pending_restart = false`, значит, сервер ещё не «узнал» об изменении. Выполняем `SELECT pg_reload_conf();` — теперь `pending_restart` станет `true`.

**Шаг 2**. Смотрим `pg_file_settings`:

```sql
SELECT name, setting, applied, error
FROM pg_file_settings
WHERE name = 'max_connections';
```

Если `applied = false` и `error` содержит "parameter cannot be changed without restarting the server", значит, параметр требует полного перезапуска кластера.

**Решение**: выполнить `sudo systemctl restart postgresql@17-main` (или `pg_ctlcluster 17 main restart`). После рестарта `SHOW max_connections` покажет 300.

## 5. Управление параметрами: best practices

1. **Используйте ALTER SYSTEM для постоянных изменений** — это безопаснее и не затирается при обновлении. Однако помните, что изменения не применяются мгновенно; всегда проверяйте `pending_restart` и `context`.

2. **Для экспериментальных изменений используйте SET** в сессии. Это позволит проверить влияние параметра без риска для всего кластера.

3. **Изменяйте параметры в правильном порядке**. Если вы, например, увеличиваете `max_connections`, не забудьте также увеличить `shared_buffers` и `work_mem` с учётом доступной ОЗУ.

4. **Перед изменением параметров, требующих перезагрузки, планируйте окно обслуживания**. Используйте `pg_settings` для проверки, какие параметры требуют рестарта.

5. **Регулярно проверяйте наличие не применённых настроек**:

   ```sql
   SELECT name, setting, sourcefile, sourceline, error
   FROM pg_file_settings
   WHERE applied = false;
   ```

   Это поможет выявить синтаксические ошибки или параметры, ожидающие перезагрузки.

6. **Используйте представления для аудита**. Запросы вида

   ```sql
   SELECT source, COUNT(*) FROM pg_settings GROUP BY source;
   ```

   покажут, насколько сильно ваша конфигурация отличается от стандартной.

7. **Документируйте изменения**. Хотя `postgresql.auto.conf` хранит историю (только последние значения), полезно вести отдельный журнал изменений параметров.

---

## Заключение

Понимание того, как работает postmaster, какие файлы конфигурации существуют и как применяются параметры, даёт вам полный контроль над PostgreSQL. Вы сможете не только настраивать сервер, но и быстро диагностировать проблемы, связанные с неправильными значениями или не применёнными изменениями.

- **Postmaster** — главный процесс, управляющий всем кластером.
- **postgresql.conf** и **postgresql.auto.conf** — основные источники настроек, причём auto.conf имеет приоритет.
- **pg_hba.conf** — ваш firewall для доступа к базам.
- **Иерархия параметров** позволяет тонко настраивать поведение для разных пользователей и баз данных.
- **pg_settings** и **pg_file_settings** — незаменимые инструменты для мониторинга и диагностики.

Помните: с большой властью приходит большая ответственность — всегда тестируйте изменения на небоевом окружении и планируйте перезагрузки заранее.
