# PostgreSQL: Резервное копирование и репликация — полное руководство с лучшими практиками

Рано или позно приходит время самого важного — **обеспечения сохранности данных и отказоустойчивости**. Вы узнаете, как правильно создавать резервные копии, восстанавливать данные и настраивать логическую репликацию между несколькими серверами. Все примеры даны с пояснениями и рекомендациями, которые используют опытные администраторы в продакшене.

---

## Часть 1. Резервное копирование и восстановление

Резервное копирование — это ваша страховка от сбоев, ошибок пользователей и злоумышленников. В PostgreSQL есть два основных подхода:

- **Логические бэкапы** — выгружают данные в виде SQL-команд или CSV-файлов. Гибкие, позволяют выборочно восстанавливать таблицы, но медленнее и не дают «моментального снимка» состояния кластера.
- **Физические бэкапы** (pg_basebackup) — копируют файлы данных целиком. Быстрее, позволяют восстановить кластер «как есть», но требуют больше места и не дают выборочного восстановления.

Мы сосредоточимся на логическом копировании, так как оно чаще используется для повседневного экспорта данных и миграций.

### 1.1. Подготовка тестовой базы

Для экспериментов создадим базу `backup_test`, схему `my_schema` и таблицу `employees`:

```sql
CREATE DATABASE backup_test;
\c backup_test
CREATE SCHEMA my_schema;

CREATE TABLE my_schema.employees (
    id SERIAL PRIMARY KEY,
    first_name VARCHAR(50),
    last_name VARCHAR(50),
    salary INTEGER
);
```

Заполним её сотней записей с помощью `generate_series`:

```sql
INSERT INTO my_schema.employees (first_name, last_name, salary)
SELECT
    'Name_' || i,
    'Surname_' || i,
    100000 + (i * 100)
FROM generate_series(1, 100) AS i;
```

Убедитесь, что данные есть:

```sql
SELECT COUNT(*) FROM my_schema.employees;   -- 100
```

### 1.2. Экспорт с помощью COPY

Команда `COPY` (или `\COPY` в psql) позволяет выгрузить данные в текстовый файл, например, в CSV. Это идеальный способ для обмена данными с другими приложениями.

Создадим каталог для бэкапов (от имени пользователя postgres):

```bash
sudo mkdir -p /var/lib/postgresql/backups
sudo chown postgres:postgres /var/lib/postgresql/backups
```

Теперь выгрузим таблицу в CSV с заголовками:

```sql
\c backup_test
\COPY my_schema.employees TO '/var/lib/postgresql/backups/employees_backup.csv' WITH CSV HEADER;
```

**Важно:** `\COPY` выполняется на стороне клиента и использует пути, доступные клиенту. Если вы подключаетесь удалённо, файл создастся на клиентской машине. Для серверного `COPY` (без обратной косой черты) пути должны быть доступны серверу.

### 1.3. Восстановление из CSV

Создадим вторую таблицу с той же структурой (без данных) и загрузим в неё CSV:

```sql
CREATE TABLE my_schema.employees2 AS TABLE my_schema.employees WITH NO DATA;
\COPY my_schema.employees2 FROM '/var/lib/postgresql/backups/employees_backup.csv' WITH CSV HEADER;
```

Проверим:

```sql
SELECT COUNT(*) FROM my_schema.employees2;   -- 100
```

**Бестпрактисы для COPY-бэкапов:**
- Используйте сжатие на уровне файловой системы (например, `gzip`) для экономии места.
- Для больших таблиц разбивайте выгрузку по диапазонам, чтобы избежать гигантских файлов.
- Всегда проверяйте целостность импортированных данных (сравнение количества записей и контрольных сумм).

### 1.4. Создание дампа с помощью pg_dump

Утилита `pg_dump` создаёт файл, содержащий SQL-команды для восстановления структуры и данных. Поддерживает разные форматы:
- **plain** (текстовый SQL) — удобен для редактирования, но не сжимается и не поддерживает выборочное восстановление.
- **custom** (сжатый, `-F c`) — сжимает данные, позволяет использовать `pg_restore` для гибкого восстановления.
- **directory** (`-F d`) — создаёт каталог с файлами, поддерживает параллельное восстановление.

Создадим дамп двух таблиц (employees и employees2) в кастомном формате с подробным выводом:

```bash
pg_dump -U postgres -d backup_test \
  -t "my_schema.employees" -t "my_schema.employees2" \
  -F c \
  -f "/var/lib/postgresql/backups/two_tables_backup.dump" \
  -v
```

Обратите внимание: если вы используете имена схем, заключайте их в двойные кавычки, как здесь.

### 1.5. Восстановление дампа с помощью pg_restore

Для кастомного формата используется `pg_restore`. Он позволяет выборочно восстанавливать объекты. Предположим, мы хотим восстановить только таблицу `employees2` в новую базу `restore_test`.

Сначала создаём базу и схему:

```sql
CREATE DATABASE restore_test;
\c restore_test
CREATE SCHEMA my_schema;
```

Пробуем восстановить конкретную таблицу с помощью флага `-t`:

```bash
pg_restore -U postgres -d restore_test -t "my_schema.employees2" /var/lib/postgresql/backups/two_tables_backup.dump -v
```

**Проблема:** Иногда `-t` не работает должным образом, потому что `pg_restore` может не восстановить таблицу, если не восстановлены зависимости (например, последовательности). В нашем случае это могло быть связано с тем, что таблицы были созданы в одной схеме, и `pg_restore` пытался восстановить схему, но не мог. Пришлось применить более надёжный метод — через файл списка.

Создаём список содержимого дампа:

```bash
pg_restore -U postgres -l /var/lib/postgresql/backups/two_tables_backup.dump > /var/lib/postgresql/backups/list.txt
```

Редактируем файл, комментируем (`#`) все строки, кроме тех, что относятся к `employees2`. Можно использовать `sed`:

```bash
sed -i '/employees2/!s/^/#/' /var/lib/postgresql/backups/list.txt
```

Теперь восстанавливаем, используя этот список:

```bash
pg_restore -U postgres -d restore_test \
  -L /var/lib/postgresql/backups/list.txt \
  /var/lib/postgresql/backups/two_tables_backup.dump -v
```

Проверяем:

```sql
\c restore_test
\dt my_schema.*
SELECT COUNT(*) FROM my_schema.employees2;   -- 100
```

**Бестпрактисы для pg_dump/pg_restore:**
- Всегда используйте кастомный формат (`-F c`) или каталог (`-F d`) для сжатия и гибкости.
- Для больших баз используйте параметр `-j` (количество параллельных потоков) при восстановлении.
- Регулярно проверяйте возможность восстановления — создавайте отдельную среду для тестовых восстановлений.
- Храните дампы на другом физическом носителе (или в облачном хранилище).
- Шифруйте бэкапы, если они содержат чувствительные данные (например, с помощью `gpg`).
- Автоматизируйте создание бэкапов через cron или систему планирования (например, `pg_cron`).

## Часть 2. Логическая репликация для масштабирования и отказоустойчивости

Репликация позволяет поддерживать копии данных на нескольких серверах. PostgreSQL поддерживает два основных типа:
- **Физическая репликация** — передача WAL-журналов на уровне байтов. Используется для создания горячих резервов (read-only).
- **Логическая репликация** — передача изменений на уровне SQL-команд (INSERT/UPDATE/DELETE). Позволяет реплицировать только определённые таблицы, между разными версиями и даже между разными СУБД (через сторонние инструменты).

В нашем примере мы настроим **логическую репликацию между тремя виртуальными машинами**, распределив нагрузку: на ВМ1 пишем в таблицу `test`, на ВМ2 — в таблицу `test2`, а на ВМ3 получаем обе таблицы для чтения и бэкапов.

### 2.1. Подготовка серверов

На каждой ВМ установлен PostgreSQL 17. Сначала на всех трёх машинах выполняем базовую настройку.

**Файл `postgresql.conf`** — разрешаем прослушивание всех интерфейсов и устанавливаем `wal_level = logical` (необходимо для логической репликации):

```ini
listen_addresses = '*'
wal_level = logical
```

Также рекомендуется увеличить значения параметров для слотов и рабочих процессов (настройте под свою нагрузку):

```ini
max_replication_slots = 10
max_logical_replication_workers = 8
max_worker_processes = 8
```

**Файл `pg_hba.conf`** — добавляем правила для подключений и репликации:

```
host    all             all             0.0.0.0/0               scram-sha-256
host    replication     all             0.0.0.0/0               scram-sha-256
```

**Устанавливаем пароль** для пользователя postgres (на всех ВМ):

```sql
ALTER USER postgres WITH PASSWORD '12345';
```

**Перезапускаем кластер** на каждой машине:

```bash
sudo pg_ctlcluster 17 main restart
```

### 2.2. Создание таблиц и публикаций

**ВМ1 (источник для таблицы test):**

```sql
-- Таблица, в которую будем писать на ВМ1
CREATE TABLE test (
    id SERIAL PRIMARY KEY,
    data VARCHAR(100),
    created_at TIMESTAMP DEFAULT NOW()
);

-- Таблица для чтения (будет получать данные с ВМ2)
CREATE TABLE test2 (
    id SERIAL PRIMARY KEY,
    data VARCHAR(100),
    created_at TIMESTAMP DEFAULT NOW()
);

-- Заполним test начальными данными
INSERT INTO test (data) VALUES ('text1'), ('text2');

-- Публикация для таблицы test
CREATE PUBLICATION test_pub FOR TABLE test;
```

**ВМ2 (источник для таблицы test2):**

```sql
CREATE TABLE test (
    id SERIAL PRIMARY KEY,
    data VARCHAR(100),
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE test2 (
    id SERIAL PRIMARY KEY,
    data VARCHAR(100),
    created_at TIMESTAMP DEFAULT NOW()
);

INSERT INTO test2 (data) VALUES ('text3'), ('text4');

CREATE PUBLICATION test2_pub FOR TABLE test2;
```

**ВМ3 (подписчик для чтения и бэкапов):**

```sql
CREATE TABLE test (
    id SERIAL PRIMARY KEY,
    data VARCHAR(100),
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE test2 (
    id SERIAL PRIMARY KEY,
    data VARCHAR(100),
    created_at TIMESTAMP DEFAULT NOW()
);
```

### 2.3. Создание подписок

Теперь связываем серверы. Подписка создаётся на стороне **подписчика** и указывает, откуда и какие публикации получать.

**На ВМ2** создаём подписку на публикацию `test_pub` с ВМ1:

```sql
CREATE SUBSCRIPTION test_sub
CONNECTION 'host=192.168.1.3 port=5432 user=postgres password=12345 dbname=postgres'
PUBLICATION test_pub;
```

**На ВМ1** создаём подписку на публикацию `test2_pub` с ВМ2:

```sql
CREATE SUBSCRIPTION test2_sub
CONNECTION 'host=192.168.1.4 port=5432 user=postgres password=12345 dbname=postgres'
PUBLICATION test2_pub;
```

**На ВМ3** создаём подписки на обе публикации:

```sql
-- Подписка на test с ВМ1
CREATE SUBSCRIPTION test_sub_vm1
CONNECTION 'host=192.168.1.3 port=5432 user=postgres password=12345 dbname=postgres'
PUBLICATION test_pub;

-- Подписка на test2 с ВМ2
CREATE SUBSCRIPTION test2_sub_vm2
CONNECTION 'host=192.168.1.4 port=5432 user=postgres password=12345 dbname=postgres'
PUBLICATION test2_pub;
```

В результате:
- ВМ1 содержит таблицы `test` (своя) и `test2` (реплика с ВМ2).
- ВМ2 содержит `test` (реплика с ВМ1) и `test2` (своя).
- ВМ3 содержит обе таблицы как реплики.

### 2.4. Проверка работы репликации

**Проверим публикации** (на ВМ1 и ВМ2):

```sql
SELECT * FROM pg_publication;
```

**Проверим подписки** (на всех ВМ):

```sql
SELECT * FROM pg_subscription;
```

**Проверим слоты репликации** (на источниках — ВМ1 и ВМ2):

```sql
SELECT slot_name, plugin, slot_type, active FROM pg_replication_slots;
```

Слоты должны быть активными (`active = t`).

**Теперь тестируем передачу данных:**

На ВМ1 вставляем запись в `test`:

```sql
INSERT INTO test (data) VALUES ('new_text_from_vm1');
```

На ВМ2 и ВМ3 проверяем, что запись появилась:

```sql
SELECT * FROM test ORDER BY id;
```

Аналогично на ВМ2 вставляем в `test2`:

```sql
INSERT INTO test2 (data) VALUES ('new_text_from_vm2');
```

Проверяем на ВМ1 и ВМ3.

**Мониторинг состояния репликации:**

- `pg_stat_replication` — показывает активные соединения репликации (физической или логической). На источниках можно увидеть подписчиков.
- `pg_stat_subscription` — статистика по подпискам на стороне подписчика, включая задержку (`last_msg_send_time`, `last_msg_receipt_time`).

```sql
-- на источнике
SELECT * FROM pg_stat_replication;

-- на подписчике
SELECT * FROM pg_stat_subscription;
```

### 2.5. Бестпрактисы для логической репликации

1. **Планирование имён.** Давайте подпискам и публикациям осмысленные имена, особенно в сложных топологиях (как в нашем примере).
2. **Мониторинг слотов.** Слоты репликации хранят WAL, пока подписчик не получит все данные. Если подписчик отключается надолго, WAL может переполнить диск. Настройте `max_slot_wal_keep_size` и отслеживайте размер слотов.
3. **Управление паролями и безопасность.** Используйте параметры подключения с SSL (`sslmode=require`) и храните пароли в файле `.pgpass` вместо указания в команде создания подписки.
4. **Обновление схемы.** Логическая репликация не реплицирует изменения схемы (ALTER TABLE). После изменения структуры таблицы на источнике нужно вручную изменить её на подписчиках (с соблюдением совместимости) или пересоздать подписку с помощью `REFRESH PUBLICATION`.
5. **Обработка конфликтов.** Если на подписчике есть данные, которые конфликтуют с вставляемыми (например, дубликат первичного ключа), подписка может остановиться. Используйте `ALTER SUBSCRIPTION ... SKIP` для пропуска конфликтующих транзакций или настройте разрешение конфликтов вручную.
6. **Тестирование отказоустойчивости.** Периодически отключайте один из серверов и проверяйте, как ведёт себя репликация при перезапуске.
7. **Резервное копирование подписчика.** ВМ3, содержащая реплики, идеально подходит для создания бэкапов без нагрузки на основные серверы. Используйте её для pg_dump или физических бэкапов.

## Заключение

Мы рассмотрели два краеугольных камня эксплуатации PostgreSQL: **резервное копирование** и **репликацию**.

- **Резервные копии** (COPY, pg_dump) — ваша последняя линия обороны. Делайте их регулярно, храните в разных местах и периодически проверяйте восстановление.
- **Логическая репликация** — мощный инструмент для распределения нагрузки, создания отдельных копий для отчётов и обеспечения географической доступности. Она проще в настройке, чем физическая, и даёт больше гибкости.

Сочетание правильных бэкапов и репликации образует надёжную систему, которая переживёт большинство сбоев. Не забывайте о мониторинге — без него вы не узнаете о проблемах вовремя. Используйте инструменты вроде `pg_stat_*`, системные метрики и alert-системы.

В следующей статье мы разберём продвинутые сценарии, такие как кластеризация с помощью Patroni и управление большими базами данных с партиционированием. А пока — экспериментируйте с репликацией, создавайте бэкапы и спите спокойно, зная, что ваши данные в безопасности.
