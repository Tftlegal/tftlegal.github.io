# PostgreSQL: Полное руководство по видам индексов для больших баз данных

Индексы — это, пожалуй, самый мощный инструмент ускорения запросов в реляционных базах данных. Без них даже простой `SELECT` по миллионной таблице может выполняться секунды, а с правильно подобранным индексом — миллисекунды. Но выбор неподходящего типа индекса или создание лишних индексов может не только не помочь, но и замедлить операции вставки и обновления. В этой статье мы разберём все типы индексов, доступные в PostgreSQL, их особенности, примеры использования и лучшие практики.

---

## 1. Зачем нужны индексы?

Индекс — это структура данных, которая позволяет быстро находить строки по значениям одного или нескольких столбцов. Без индекса PostgreSQL выполняет **последовательное сканирование** (Seq Scan) всей таблицы, что на больших объёмах данных катастрофически медленно.

Индексы особенно полезны для:
- Условий `WHERE`
- Сортировки (`ORDER BY`)
- Соединений (`JOIN`)
- Ограничений уникальности (`UNIQUE`)
- Полнотекстового поиска

Однако у индексов есть цена:
- Они занимают место на диске (иногда больше, чем сама таблица)
- Замедляют `INSERT`, `UPDATE`, `DELETE`, так как индекс нужно обновлять при каждом изменении данных

Поэтому индексы нужно создавать осознанно, только под реальные запросы.

---

## 2. Основные типы индексов в PostgreSQL

PostgreSQL поддерживает несколько типов индексов, каждый из которых оптимизирован для определённых операций и типов данных.

| Тип индекса | Поддерживаемые операции | Когда использовать |
|-------------|-------------------------|-------------------|
| **B-tree** | `=`, `>`, `<`, `>=`, `<=`, `BETWEEN`, `IN`, `LIKE` (без ведущего `%`) | Стандартный выбор для большинства сценариев. Работает с любыми типами данных. |
| **Hash** | `=` | Только для точного равенства. Может быть быстрее B-tree, но не поддерживает сортировку и диапазоны. Используется редко. |
| **GiST** | Геометрические операторы, полнотекстовый поиск, поиск по диапазонам, `LIKE` с `%` | Для нетрадиционных типов данных: геоданные, массивы, полнотекст, запросы на расстояние. |
| **SP-GiST** | Точки, диапазоны, IP-адреса, полнотекст (отличается от GiST) | Для данных с естественной кластеризацией (например, дерево квадрантов для точек). |
| **GIN** | Массивы, JSONB, полнотекстовый поиск, `LIKE` с `%`, поиск по нескольким значениям | Для составных значений (массивы, JSON) и полнотекстового поиска. Очень эффективен для «перевёрнутых индексов». |
| **BRIN** | Диапазоны, сортировка, большие таблицы с физической упорядоченностью | Для очень больших таблиц (миллиарды строк), где столбец естественно коррелирует с физическим расположением (например, `timestamp`). Занимает мало места. |

---

## 3. B-tree — универсальный солдат

**B-tree** (сбалансированное дерево) — индекс по умолчанию в PostgreSQL. Он используется, если тип индекса не указан явно. Поддерживает все операторы сравнения, сортировку и уникальность.

### 3.1. Создание B-tree индекса

```sql
CREATE INDEX idx_passenger_name ON tickets (passenger_name);
```

Этот индекс ускорит запросы вида:

```sql
SELECT * FROM tickets WHERE passenger_name = 'Natalya Filippova';
SELECT * FROM tickets WHERE passenger_name LIKE 'Natalya%';
SELECT * FROM tickets ORDER BY passenger_name;
```

### 3.2. Уникальный B-tree индекс

```sql
CREATE UNIQUE INDEX idx_unique_book_ref ON tickets (book_ref);
```

Или через ограничение `UNIQUE`:

```sql
ALTER TABLE tickets ADD CONSTRAINT tickets_book_ref_unique UNIQUE (book_ref);
```

Уникальный индекс не только ускоряет поиск, но и гарантирует отсутствие дубликатов.

### 3.3. Анализ эффективности

Созданный индекс на `passenger_name` сократил время выполнения запроса с 340 мс до 0.15 мс. В плане выполнения виден **Index Scan**:

```
Index Scan using idx_passenger_name on tickets
  (cost=0.43..9.57 rows=8 width=52)
  (actual time=0.082..0.125 rows=17 loops=1)
  Index Cond: (passenger_name = 'Natalya Filippova'::text)
```

- `cost` — оценка стоимости (чем меньше, тем лучше)
- `actual time` — реальное время выполнения
- `Index Cond` — условие, использованное при сканировании

Если бы индекса не было, мы бы увидели `Seq Scan`.

## 4. GIN — для полнотекстового поиска и составных данных

**GIN** (Generalized Inverted Index) — индекс для составных значений, таких как массивы, JSONB, и для полнотекстового поиска. Он строит «перевёрнутый» индекс, где ключи — это отдельные элементы, а значения — ссылки на строки.

### 4.1. Полнотекстовый индекс на русском языке

В конспекте создан индекс:

```sql
CREATE INDEX idx_passenger_name_gin ON tickets
USING gin (to_tsvector('russian', passenger_name));
```

Теперь можно искать по частям имени с использованием операторов `&` (И), `|` (ИЛИ), `!` (НЕ):

```sql
SELECT * FROM tickets
WHERE to_tsvector('russian', passenger_name) @@ to_tsquery('russian', 'Natalya & Filippova');
```

Этот запрос найдёт все билеты, где имя содержит и «Natalya», и «Filippova». Без GIN-индекса такой поиск выполнялся бы полным сканированием.

**Важно:** `to_tsvector` создаёт лексемы (слова) с учётом стемминга (приведения к нормальной форме). Для русского языка нужно установить конфигурацию `'russian'`. Если вы используете английский, замените на `'english'`.

### 4.2. GIN для JSONB

```sql
CREATE INDEX idx_json_data ON orders USING gin (data);
```

Ускоряет запросы с операторами `?`, `?&`, `?|` и `@>`:

```sql
SELECT * FROM orders WHERE data @> '{"status": "shipped"}';
```

### 4.3. GIN для массивов

```sql
CREATE INDEX idx_tags ON articles USING gin (tags);
```

```sql
SELECT * FROM articles WHERE tags @> ARRAY['postgresql', 'index'];
```

### 4.4. Расширение pg_trgm для нечёткого поиска

Для поиска по частичному совпадению с `LIKE '%text%'` эффективнее использовать `pg_trgm`:

```sql
CREATE EXTENSION pg_trgm;
CREATE INDEX idx_passenger_name_trgm ON tickets USING gin (passenger_name gin_trgm_ops);
```

Теперь запросы с `LIKE '%lipp%'` будут использовать индекс.

## 5. Частичные индексы (Partial Index)

Частичный индекс содержит только строки, удовлетворяющие условию `WHERE`. Он экономит место и ускоряет запросы к определённым подмножествам данных.

### 5.1. Пример из конспекта

```sql
CREATE INDEX idx_partial_outbound_name ON tickets (passenger_name)
WHERE outbound = true;
```

Этот индекс пригодится для запросов:

```sql
SELECT * FROM tickets WHERE outbound = true AND passenger_name = 'Natalya Filippova';
```

В конспекте отмечается, что этот индекс оказался **менее эффективным**, чем простой индекс на `passenger_name`, потому что планировщик предпочёл уже существующий индекс `idx_passenger_name`, который покрывает все строки. Если же у вас много запросов именно к `outbound = true`, и они составляют лишь малую часть таблицы, частичный индекс может быть очень полезен.

### 5.2. Когда использовать частичные индексы

- Когда часто фильтруют по значению, которое встречается редко (например, `status = 'active'` в таблице с архивными записями)
- Когда нужно исключить `NULL` значения, чтобы уменьшить размер индекса
- Для обеспечения уникальности только для определённого подмножества (например, уникальный email только для активных пользователей)

```sql
CREATE UNIQUE INDEX idx_active_user_email ON users (email) WHERE is_active = true;
```

## 6. Многоколоночные индексы (Composite Index)

Индекс на нескольких колонках ускоряет запросы, которые фильтруют или сортируют по нескольким столбцам одновременно.

### 6.1. Пример из конспекта

```sql
CREATE INDEX idx_multiple_columns ON tickets (book_ref, passenger_id, outbound);
```

Однако в конспекте этот индекс **работал медленнее**, потому что уже существовал уникальный индекс `tickets_book_ref_passenger_id_outbound_key`, который дублировал эту комбинацию. PostgreSQL может использовать уже существующий индекс, и новый только добавляет накладные расходы.

### 6.2. Правила составления многоколоночных индексов

- **Порядок столбцов важен**: индекс может использоваться для условий на левый столбец, на левый+второй, на все три, но не на второй отдельно без первого.
- Для условий `=`, `>` и `ORDER BY` лучше ставить столбцы с наибольшей селективностью первыми.
- Если запрос фильтрует по одному столбцу, а сортирует по другому, включите оба.

Пример правильного порядка:

```sql
CREATE INDEX idx_book_date ON bookings (book_date, total_amount);
-- Ускоряет:
SELECT * FROM bookings WHERE book_date = '2025-10-01' ORDER BY total_amount;
```

### 6.3. Индекс на `(a, b)` vs два отдельных индекса `(a)` и `(b)`

Иногда лучше создать два отдельных индекса, так как PostgreSQL может объединять их через Bitmap And/Or. Но для точных условий на оба столбца составной индекс будет эффективнее.

## 7. Индексы по выражениям (Expression Index)

Позволяют индексировать результат функции или выражения, а не само значение столбца.

### 7.1. Пример

```sql
CREATE INDEX idx_lower_name ON users (LOWER(name));
```

Теперь запросы с `WHERE LOWER(name) = 'john'` будут использовать индекс.

### 7.2. Индекс для полнотекстового поиска

По сути, `to_tsvector` — это выражение, поэтому такой индекс тоже является индексом по выражению.

### 7.3. Когда использовать

- Для поиска без учёта регистра
- Для поиска по производным полям (например, из JSONB)
- Для вычисляемых полей, которые часто используются в `WHERE`

---

## 8. BRIN — индекс для гигантских таблиц

**BRIN** (Block Range Index) хранит не значения, а метаданные о блоках данных (минимальное и максимальное значение в каждом блоке). Он очень компактен (килобайты на гигабайты данных) и эффективен для колонок, значения которых коррелируют с физическим расположением (например, `timestamp`, `serial`).

### 8.1. Создание BRIN-индекса

```sql
CREATE INDEX idx_bookings_date_brin ON bookings USING BRIN (book_date);
```

### 8.2. Когда использовать

- Очень большие таблицы (миллиарды строк)
- Столбец имеет высокую корреляцию с физическим порядком
- Запросы выбирают узкий диапазон значений (например, `WHERE book_date BETWEEN '2025-10-01' AND '2025-10-02'`)

### 8.3. Сравнение с B-tree

Для таблицы на 100 млн строк B-tree индекс на `timestamp` может занимать несколько гигабайт, а BRIN — всего несколько мегабайт. При этом BRIN даёт ускорение, близкое к B-tree, если запросы точечные по диапазону.

---

## 9. GiST и SP-GiST — для пространственных и сложных данных

**GiST** (Generalized Search Tree) — универсальный индекс для нетрадиционных данных: геометрических, полнотекстовых, диапазонов и т.д.

**SP-GiST** (Space-Partitioned GiST) — оптимизирован для данных с естественным разбиением пространства (например, точки на плоскости).

### 9.1. Пример для геоданных

```sql
CREATE EXTENSION postgis;
CREATE INDEX idx_geo ON locations USING gist (geom);
```

Ускоряет запросы на расстояние:

```sql
SELECT * FROM locations WHERE ST_DWithin(geom, ST_MakePoint(30, 40), 1000);
```

### 9.2. Индекс на диапазоны

```sql
CREATE INDEX idx_daterange ON events USING gist (period);
```

```sql
SELECT * FROM events WHERE period @> '2025-10-01'::date;
```

---

## 10. Hash — только для равенства

Индекс Hash используется редко, так как B-tree лучше подходит для равенства и поддерживает больше операций. Однако в некоторых случаях Hash может быть немного быстрее для `=`.

```sql
CREATE INDEX idx_hash_id ON users USING hash (id);
```

Но в PostgreSQL 10+ Hash-индексы стали crash-safe, но всё равно уступают B-tree по универсальности.

---

## 11. Создание индексов без блокировок (CONCURRENTLY)

В production-среде создание индекса обычным `CREATE INDEX` блокирует таблицу на запись (но не на чтение) на всё время создания. Для больших таблиц это может длиться часы и приводить к простою приложения.

### 11.1. Использование CONCURRENTLY

```sql
CREATE INDEX CONCURRENTLY idx_passenger_name ON tickets (passenger_name);
```

- Индекс создаётся в фоновом режиме без блокировки записи
- Может занять больше времени
- Если создание прервётся, остаётся «недействительный» индекс, который нужно удалить

### 11.2. Удаление индекса без блокировок

Удаление индекса всегда происходит без блокировок, поэтому специального ключа не требуется.

## 12. Мониторинг использования индексов

Создать индекс — полдела. Важно отслеживать, используется ли он, и не создаёт ли он лишней нагрузки.

### 12.1. Статистика использования индексов

```sql
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan,          -- количество сканирований индекса
    idx_tup_read,      -- прочитано строк
    idx_tup_fetch,     -- возвращено строк
    pg_size_pretty(pg_relation_size(indexname)) AS index_size
FROM pg_stat_user_indexes
ORDER BY idx_scan DESC;
```

### 12.2. Поиск неиспользуемых индексов

```sql
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY pg_relation_size(indexname) DESC;
```

Если индекс не использовался ни разу, его лучше удалить.

### 12.3. Поиск дублирующихся индексов

Можно использовать расширение `pg_stat_statements` и `pg_indexes`, чтобы найти похожие индексы.

---

## 13. Удаление индексов

```sql
DROP INDEX idx_passenger_name;
```

Для удаления недействительного индекса, оставшегося после неудачного `CREATE INDEX CONCURRENTLY`, используйте `DROP INDEX CONCURRENTLY` (если нужно избежать блокировок).

---

## 14. Best Practices по индексам

1. **Начинайте с анализа запросов** — используйте `EXPLAIN` и `pg_stat_statements`, чтобы найти самые медленные запросы.
2. **Создавайте индексы под конкретные условия** — не индексируйте все столбцы подряд.
3. **Используйте `CONCURRENTLY` в production** — чтобы не останавливать работу приложения.
4. **Регулярно проверяйте использование индексов** и удаляйте неиспользуемые.
5. **Частичные индексы** — отличный способ сэкономить место, если часто ищут по определённому значению.
6. **Для полнотекстового поиска** используйте GIN или GiST, а не B-tree.
7. **Для очень больших таблиц с временными столбцами** рассмотрите BRIN вместо B-tree.
8. **Многоколоночные индексы** создавайте с правильным порядком столбцов (слева направо по убыванию селективности).
9. **Уникальные индексы** — не только для ограничений, но и для ускорения поиска.
10. **Мониторьте размер индексов** — если индекс больше таблицы, возможно, он неоптимален.
11. **Тестируйте на staging** перед внедрением в production.

---

## 15. Заключение

PostgreSQL предлагает богатый набор типов индексов, позволяющий оптимизировать практически любой сценарий. B-tree — ваш основной инструмент, но для специфических задач (полнотекст, JSONB, массивы, геоданные, гигантские таблицы) существуют специализированные индексы: GIN, GiST, BRIN и другие. Правильный выбор типа индекса в сочетании с грамотным мониторингом и обслуживанием превратит вашу базу данных в высокопроизводительную систему.

Не забывайте: индексы — это не панацея. Иногда лучше переписать запрос, изменить схему или использовать материализованные представления. Всегда анализируйте планы выполнения и тестируйте изменения. 
И помните о балансе: каждый новый индекс ускоряет чтение, но замедляет запись.
